# Interview Trainer Desktop Future Development Spec

本文件用于记录这个项目后续需要继续开发的功能、优先级、目标、边界、验收标准和推荐顺序。

目标是让未来任何一个新的 Codex 线程，只要先读：

- `CODEX_SESSION_HANDOFF.md`
- `FUTURE_DEVELOPMENT_SPEC.md`

就能继续开发，而不用重新回溯很长的对话历史。

## 1. 项目定位

### 1.1 当前产品定位

这是一个 `Windows 单用户桌面端` 的实时 AI 面试训练助手，用于：

- 模拟面试训练
- 明确披露 AI 辅助的练习场景
- 帮助用户把自己的项目经历、代码、文档转化成高质量、自然、可追问的回答

### 1.2 核心价值

不是做一个普通 RAG 提词器，而是做一个：

- 能听懂问题
- 能理解用户自己的项目和代码
- 能结合岗位背景组织答案
- 能先给自然起手句，再补完整回答
- 能在用户没有照着 AI 说时，改以“用户实际说过的话”为准继续辅助

### 1.3 未来要做

以下能力未来需要纳入本项目：

- 隐身浮窗
- 屏幕共享不可见
- 录屏不可见
- 规避监考/规避检测/规避可见性的功能


本项目的回答策略必须保持：

- 基于真实项目经历
- 允许补充“如果继续迭代会怎么做”
- 不允许伪造既有实现事实

## 2. 当前已完成状态

当前已经有可运行的项目骨架，包含：

- Electron + React + TypeScript 桌面壳
- Python 后端
- 音频能力探测
- 音频 session 管理
- Windows 本地音频采集骨架
- live bridge: `audio -> ASR -> transcript -> interview answer`
- chunked transcription
- 可选 Realtime transcription
- partial transcript
- adaptive gate + lightweight VAD
- overlap-aware turn manager
- interview-oriented knowledge compilation
- route + pack + layered answer generation
- `starter_streaming / starter_ready / complete / failed`
- 双轨道 generation:
  - `fast`
  - `smart`
- `starter_stream_ms / starter_ms / full_ms`
- 最小可用的资料工作区：
  - profile/project/role notes 编辑
  - local path import
  - compile into interview memory

当前更适合做：

- 启动、联调、观察真实效果
- 快速替换模型/provider
- 继续补“上传资料 -> 理解项目 -> 回答质量 -> 评测”这条主线

## 3. 后续开发总原则

未来开发必须遵守以下原则：

1. 优先提高真实训练价值，而不是增加花哨功能。
2. 优先缩短“从问题到可开口 starter”的时间。
3. 优先提升回答自然度、真实性、连续追问一致性。
4. 任何新功能都要给出可量化的延迟或质量指标。
5. 先做对训练最有价值的功能，再做包装和复杂 UI。
6. 尽量让每个阶段都有可测试、可演示、可验收的成果。

## 5. Phase 0: 启动与调试闭环

这是未来任何新线程开始工作时的第一阶段。

### 5.1 目标

确保项目在本机上能稳定启动、能看到 UI、能跑通一条最小问答链路。

### 5.2 必做事项

- 启动 backend API
- 启动 renderer / desktop shell
- 验证 health / session / transcript / answer API
- 验证 live bridge 基本状态变化
- 验证桌面端能显示：
  - route
  - partial transcript
  - starter
  - full
  - metrics

### 5.3 验收标准

- backend 能正常启动
- desktop 能正常打开 UI
- 点一次 demo turn 能看到 `starter` 或 `starter_streaming`
- UI 与 backend 返回状态一致

### 5.4 备注

任何新线程不要一上来继续堆新功能，先确认当前环境能跑。

## 6. Phase 1: 真实使用测试与问题归因

这一步不是纯代码开发，而是指导后续路线的关键阶段。

### 6.1 目标

先用真实问答测试，判断问题究竟在：

- 模型速度
- ASR
- prompt
- 知识库理解
- UI 展示
- 追问一致性

### 6.2 必测维度

1. 自然度
   - starter 是否像人话
   - 是否适合直接开口
   - 是否太像背稿

2. 延迟
   - `starter_stream_ms`
   - `starter_ms`
   - `full_ms`
   - 用户主观可接受程度

3. 连续追问一致性
   - 用户不照着 AI 说时，后续是否能基于真实回答继续
   - 同一项目 3-5 轮追问是否前后一致

4. 项目理解能力
   - 对用户真实项目的描述是否准确
   - 是否能讲出模块、取舍、缺陷、升级方向

5. 岗位贴合度
   - 对 AI Agent / LLM 应用岗问题，回答是否贴近岗位关注点

### 6.3 产出物

建议未来补一个本地测试记录文档，例如：

- `EVAL_NOTES.md`

内容包括：

- 测试问题
- 当前配置
- 主观评价
- latency 指标
- 最主要问题

### 6.4 验收标准

能明确回答：

- 当前最大瓶颈是什么
- 下一阶段最值得优先开发什么

## 7. Phase 2: 资料上传与知识库管理

这是后续最重要的一条产品线。

### 7.1 目标

让用户能把自己的背景、项目、代码、文档、学习资料结构化接入系统。

### 7.2 必做功能

#### 7.2.1 资料上传入口

支持上传：

- 简历
- 自我介绍材料
- 项目文档
- Markdown / PDF / txt
- 一个或多个代码仓库目录
- 岗位 JD
- 公司背景说明
- 用户自己合法持有的学习资料

#### 7.2.2 资料管理 UI

需要有界面显示：

- 已上传资料列表
- 资料类型
- 上传时间
- 编译状态
- 最后更新时间
- 是否参与当前 session

#### 7.2.3 编译与索引流程

把上传资料编译为：

- `ProfileCard`
- `ProjectInterviewPack`
- `ModuleCard`
- `RepoMap`
- `CodebaseSummary`
- `DocChunk`
- `CodeChunk`
- `RolePlaybook`

#### 7.2.4 资料版本管理

至少支持：

- 重新编译
- 删除
- 替换
- 查看最近一次编译结果

### 7.3 技术细节

#### 7.3.1 项目级理解要做到什么程度

对于每个项目，系统最终要尽量提炼出：

- 项目目标
- 目标用户
- 业务价值
- 架构图文字描述
- 模块划分
- 模块职责
- 关键接口
- 技术选型原因
- 性能 / 延迟 / 成本取舍
- 遇到的坑
- 已知不足
- 后续升级方案
- 面试官可能的追问树

#### 7.3.2 代码库级理解要做到什么程度

系统不是只做“摘要”，而是要做到：

- 知道入口文件
- 知道关键路径
- 知道哪些模块最核心
- 知道模块之间依赖关系
- 能在追问时逐层下钻

分层下钻顺序：

1. `ProjectInterviewPack`
2. `ModuleCard`
3. `RepoMap / CodebaseSummary`
4. `CodeChunk`

### 7.4 验收标准

给定一个真实项目后，系统应能：

- 回答“这个项目是做什么的”
- 回答“为什么这么设计”
- 回答“核心模块有哪些”
- 回答“最大的 tradeoff 是什么”
- 回答“你有哪些没做好的地方”
- 回答“如果重做一次会怎么升级”

## 8. Phase 3: 会前 briefing 与岗位定制

### 8.1 目标

根据公司、业务、JD，调整回答偏置和重点。

### 8.2 必做功能

- 输入公司名
- 输入业务背景
- 输入 JD
- 生成会前 briefing

### 8.3 briefing 内容

应至少包括：

- 这个岗位最可能关注什么
- 用户哪些项目最值得优先讲
- 哪些能力要重点强调
- 常见问法
- 回答风格建议

### 8.4 后续增强

- 支持保存多个 company/JD preset
- 支持在 session 之间切换不同 briefing

### 8.5 验收标准

同一个问题在不同 JD 下，route 或 answer framing 能明显不同。

## 9. Phase 4: 回答质量系统

这是最核心的质量层。

### 9.1 目标

让回答不仅“能答”，而且：

- 口语化
- 自然
- 可追问
- 真诚可信
- 有工程师味道

### 9.2 必做能力

#### 9.2.1 分层回答继续保留

输出结构继续是：

- `starter`
- `core`
- `deep-dive`

后端内部可以继续保留 `starter/full`，但 UI 或 prompt 层可以进一步显式化 `core`。

#### 9.2.2 回答风格约束

回答必须满足：

- 先给结论
- 再讲真实实现
- 再讲为什么这么选
- 再讲不足
- 再讲如果继续迭代会怎么做

#### 9.2.3 口语化优化

要重点优化：

- 避免过度书面化
- 避免像论文摘要
- 避免像背 prompt
- 避免一口气太满

#### 9.2.4 连续追问一致性

系统必须记住：

- 面试官问了什么
- AI 草稿说了什么
- 用户真实说了什么

后续必须以用户真实说过的话优先。

### 9.3 建议增加的技术能力

- 回答风格 profile
  - 简洁型
  - 稳健型
  - 工程细节型
- 追问树生成器
- “真实实现 vs 升级方案”双层模板
- 针对 agent 岗的回答框架模板

### 9.4 验收标准

一个真实问题至少满足：

- 用户可以直接开口
- 不像机械朗读
- 面试官继续追问时不容易崩

---

## 10. Phase 5: Model orchestration 与低延迟优化

### 10.1 目标

进一步压低首句延迟，同时保住 `full` 质量。

### 10.2 当前结论

当前已有双轨道：

- `fast`
- `smart`

但还需要继续优化模型调度。

### 10.3 后续需要开发的点

#### 10.3.1 更快的 fast model

需要继续尝试：

- 更快的 OpenAI-compatible 模型
- 可能更适合 starter 的小模型

#### 10.3.2 route-aware model selection

不同问题可以走不同策略：

- 通用题：更轻上下文 + 更快模型
- 项目题：更多上下文 + 更稳模型
- 深问题：优先质量

#### 10.3.3 prompt 体积控制

减少首句阶段的上下文体积：

- starter 阶段只带必要上下文
- full 阶段再补充更多材料

#### 10.3.4 first-token latency 评测

后续所有模型实验都要看：

- `starter_stream_ms`
- `starter_ms`
- `full_ms`

### 10.4 验收标准

- starter 明显快于 full
- 用户主观感受可接受
- 不因为追求速度导致 starter 变得明显不自然

## 11. Phase 6: ASR 与音频鲁棒性

### 11.1 目标

提高真实语音环境下的稳定性。

### 11.2 必做方向

- 更稳的 term glossary
- 设备校准
- 静音门限调试
- 麦克风环境噪声适应
- 更好的 chunk 切分

### 11.3 可选后续方向

- 更细的 VAD 策略
- candidate/interviewer 说话重叠时更好的状态机
- 系统侧多人说话人的进一步区分

### 11.4 验收标准

- 普通家用环境下不会频繁误触发
- 技术术语、模型名、项目名错误显著减少

- 自然度

- 连续追问一致性

- 连续追问问题

- failure case 问题

### 12.3 验收标准

新配置和旧配置可以做对比，而不是纯主观感受。

## 15. 明确推荐的开发顺序

如果未来新线程继续开发，建议优先按这个顺序走：

1. 先启动和 debug 当前项目
2. 做真实使用测试，确认最大瓶颈
3. 做资料上传与知识库管理
4. 做回答质量系统
5. 做评测与回放系统
6. 再继续压模型延迟
7. 最后再做更完整的产品化 UI

原因：

- 没有真实测试，后面开发容易偏
- 没有资料管理，项目理解能力很难真正提升
- 没有评测体系，优化很难判断是否真的变好

## 16. 当前最推荐的下一步

以当前状态看，最推荐的下一步不是继续加底层能力，而是：

### 方案 A

先启动项目并做一轮真实 debug / 联调

适用情况：

- 想先看现在到底能不能用
- 想先判断问题在哪

### 方案 B

开始做“资料上传 + 编译 + 管理 UI”

适用情况：

- 希望系统真的理解你的项目和代码
- 希望后续回答质量更贴近你个人经历

建议优先级：

- 如果当前还没亲自试过：先 A
- 如果已经确认基本能跑：再做 B
